# Event Sourcing & FlowEvents

ChatCore is **event-sourced**: every action — a message, an edit, a membership
change, a topic change — is recorded as an immutable `FlowEvent`. Current room
state can be derived by replaying events, and ChatCore also maintains a small
room-state projection so common state reads do not need to replay the whole
timeline.

## The FlowEvent

A `FlowEvent` is the atomic, immutable unit of state. Once written, it is never
updated or deleted. Each event carries:

- **`id`** — a unique identifier generated by the storage backend.
- **`roomId`** — the conversation boundary the event belongs to.
- **`senderId`** — the entity that initiated the event.
- **`type`** — a dot-notated namespace, e.g. `message.text` or
  `room.state.name`.
- **`content`** — a JSON-serializable payload, schema-less by design.
- **`sequenceId`** — a globally monotonic cursor assigned at publish time (see
  [Sync](./sync.md)).

## Why immutability

Because events are never mutated, ChatCore gets several properties for free:

- **Auditability** — the full history of a room is always recoverable.
- **Time travel** — you can reconstruct a room's state as of any `sequenceId`.
- **Conflict-free replication** — appending is the only write operation.

## Publishing events

Append an event with `publishEvent`. It returns the stored event together with
the `sequenceId` assigned to it:

```ts
const { event, sequenceId } = await flow.publishEvent({
  roomId: room.id,
  senderId: "u1",
  type: "message.text",
  content: { body: "hello" },
});
```

Edits and reactions are *also* events — they reference the events they act on
via `parentEventIds`, forming a [threaded DAG](./rooms-and-state.md).
